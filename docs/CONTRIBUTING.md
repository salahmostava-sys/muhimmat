# Contributing — قواعد الصيانة والمراجعة

> الهدف: كود **يُراجع بسهولة** و**يُوسَّع** بدون تداخل أو تكرار.

---

## 1) قبل أي تعديل كبير

- [ ] `npm run build` (من الجذر أو `frontend/`) يمر بدون أخطاء.
- [ ] `npm run lint` (أو على الأقل الملفات التي لمستها).
- [ ] إن وُجدت اختبارات ذات صلة: `npm run test`.

---

## 2) أين أضع الكود الجديد؟

| نوع التغيير | المكان |
|-------------|--------|
| استدعاء جدول / RPC / منطق DB | `frontend/services/<domain>Service.ts` |
| منطق جلب يُعاد استخدامه + React Query | `frontend/shared/hooks/useXxx.ts` |
| صفحة أو مسار جديد | `frontend/modules/...` + تسجيل في `frontend/app/App.tsx` |
| مكون UI عام | `frontend/shared/components/` |
| سياقات عامة (auth, لغة) | `frontend/app/providers/` |

---

## 3) قواعد البيانات (React Query)

- **Query keys:** واضحة، ثابتة قدر الإمكان؛ تضمين `userId` أو معاملات المسار عند الحاجة.
- **`enabled`:** لا تشغّل `useQuery` بدون `user?.id` أو المعرفات المطلوبة إن وُجدت.
- **بعد التعديلات:** `invalidateQueries` أو `refetch` حيث يلزم — لا تعتمد على “إعادة تحميل الصفحة” كحل وحيد.

---

## 4) قواعد الأخطاء

- في **services**: استخدم `handleSupabaseError` (أو `toServiceError`) كما هو متفق عليه في الملف؛ لا تبتلع الخطأ بصمت.
- في **UI:** معالجة الأخطاء للمستخدم (toast / رسالة)؛ `console.error` للمساعدة في التشخيص فقط وليس بديلاً عن رسالة للمستخدم في العمليات الحرجة.

---

## 5) TypeScript والأسماء

- تجنّب `any` إلا حيث ضرورة مؤقتة؛ وثّق السبب.
- أسماء الدوال والمتغيرات: **إنجليزي** ووصفية (مشروع العمل بالعربية في الواجهة).

---

## 6) Git

- رسائل commit: جملة واحدة واضحة (مثلاً `fix:`, `feat:`, `chore:`).
- لا ترفع: `node_modules/`, `dist/`, `frontend/coverage/`, ملفات بيئة سرية.
- **PR على GitHub:** يُملأ الوصف تلقائياً من **`.github/pull_request_template.md`** — أكمل الأقسام قبل طلب المراجعة.
- **تغيير يهم المستخدمين أو الصيانة بشكل واضح:** أضف سطراً تحت **`[Unreleased]`** في **`CHANGELOG.md`** (في نفس الـ PR عند الإمكان).

---

## 7) المراجعة (Code review)

يراجع المُراجع:

- هل التغيير في **الطبقة الصحيحة** (خدمة vs صفحة)?
- هل هناك **تكرار** يمكن دمجه في service أو hook?
- هل **الصلاحيات/RLS** على الخادم ما زالت صحيحة لهذا المسار?

---

*هذا الملف يُحدَّث مع نضج المشروع؛ اقتراحات التحسين تُرحّب بها.*
